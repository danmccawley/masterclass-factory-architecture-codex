# Masterclass Factory — Agent Architecture, Scaffold & Build Plan

**Purpose.** A portable build contract for the *generation system* that turns a topic + intake
form into a deployable interactive masterclass matching the existing engine. Use this in Claude,
Claude Code, Codex, and ChatGPT as the single source of truth.

**Companion doc.** This sits on top of *"Masterclass Engine — Architecture & Feature Scaffold"*
(the engine build contract). That doc defines the **runtime** (engine, shell, backends, data
contracts). This doc defines the **Factory** (the agents that author the content layer those
contracts expect). Read that one first; this one assumes it.

---

## 0. TL;DR — the three decisions

1. **Build a multi-agent *pipeline with gates*, not a swarm and not a monolith.** The work
   decomposes into stages with different cognitive modes, different tools, and two hard
   correctness gates. Run it as an ordered pipeline where the Orchestrator routes work and
   enforces gates. (Decision rule in §2.)

2. **The Factory only ever produces the *content layer*.** Engine, shell, and backends are
   *fixed templates copied unchanged*. The Factory generates `content.js`
   (`SLIDES`/`POLLS`/`WORDS`), `glossary.js` (`GLOSSARY`), `source.js` (`SOURCE_PAPER`), and de-topics
   a handful of strings. Content is data, not code.

3. **The intake form compiles to one artifact: the *Course Brief* JSON (§4).** Every agent reads
   from the Brief. The 8 UX steps exist only to produce a valid Brief. Get the Brief right and the
   rest is mechanical.

**How to use this across tools.** The agent roster (§5) is written so each agent is *either* a
Claude Code subagent / skill, *or* a sequential prompt stage in ChatGPT/Codex, *or* a node in an
API orchestration script — same role, same inputs/outputs, same done-criteria. Tool-specific notes
in §11.

## 1. What's fixed vs. what's generated

| Layer | Fixed (copy unchanged) | Generated by the Factory |
|---|---|---|
| **Engine** | `engine.js`, `navscrubber.js` | — |
| **Shell** | `index.html` (CSS tokens, modal markup, load order) | the ~5 topic strings inside it (chat-modal desc, etc.) |
| **Backends** | `api/chat.js`, `api/grade.js`, `api/tts.js`, `api/poll.js`, `api/words.js`, `api/feedback.js`, `api/quality.js` | the 2 system-prompt topic strings in `chat.js` + `grade.js` |
| **Content** | — | `content.js` (`SLIDES`,`POLLS`,`WORDS`), `glossary.js` (`GLOSSARY`), `source.js` (`SOURCE_PAPER`) |
| **Presenter** | `build_script.js` (docx generator) | re-run after content to emit the Facilitator script |

> **Contract — VERIFIED against both shipped engines (MDC + Texas).** Both engines read
> `window.SOURCE_PAPER`, **not** an array. The Texas (newer) engine also reads deck-defined
> `window.POLLS` / `window.WORDS` (empty `{}` fallback) — `POLLS`/`WORDS` are only hardcoded in the
> *older* MDC engine. The source contract is:
> ```js
> window.SOURCE_PAPER = { title, cite, sections:[ { id:"s1", num, title, body }, … ] };
> ```
> Citations `<sup class="cite" data-src="s7">` resolve to a `sections[].id` — so the Works-Cited /
> "Student Reader" is **one paper object with a `sections[]` array**, not a flat `SOURCES[]`. A flat
> `SOURCES[]` would require rewriting `renderSource`/`openSource` in the engine — don't, unless you
> mean to. Sanity-check per engine version with
> `grep -nE "window\.(SLIDES|POLLS|WORDS|SOURCE_PAPER|GLOSSARY)" engine.js`.

## 2. Architecture decision: multi-agent, and exactly how much

**Recommendation: an orchestrated pipeline of specialized agents with two independent gates.**

**Why multi-agent (not one big agent):**
- The stages need genuinely different things: research wants retrieval + web tools; authoring wants
  long-form generation; codegen wants determinism; QA wants a code sandbox. One context juggling all
  of it degrades on each.
- **Two stages must be *independent* — i.e., run by an agent that did not produce the work it's
  judging** — because they are correctness gates, and a context that wrote the content will rubber-stamp
  it:
  - **Source Verification** (anti-hallucination; "no fabricated/unverified citations" is non-negotiable).
  - **QA / Verifier** (the parse-and-count harness; the Definition of Done).
- Long content parallelizes cleanly **by section**, which a multi-agent layout exploits.

**Why *not* a free-roaming swarm:**
- The pipeline is naturally ordered (Brief → corpus → plan → content → checks → build → deploy).
  Ordering + gates beats emergent coordination here, and is far easier to debug and to reproduce
  across four different tool ecosystems.

**The decision rule (use this to avoid over-engineering):**
> Start with **one agent that plays every role in sequence** for the MVP — *except* Source
> Verification and QA, which are **always** separate, independent passes. Split a role into its own
> agent only when (a) it needs a tool the others don't, (b) it's a correctness gate, or (c) you want
> to parallelize it. A 12-slide internal explainer with no outside research can be a single agent +
> the two gates. A 90-slide researched deck wants the full roster.

## 3. The pipeline (data flow)

```
[1] Intake/Brief ──► Course Brief JSON (§4)  ── soft gate: human sign-off
        │
[2] Knowledge Base / Research ──► Cited Corpus (claims + provenance)  ── GATE: credibility
        │
[3] Curriculum / Learning Design ──► Lesson Plan (sections, slide budget, interaction map, level plan)
        │
        ├──[4] Content Author (section-parallel) ──► slide decks + deep-dive papers
        ├──[5] Glossary ──► GLOSSARY (terms actually used)
        └──[6] Assessment ──► quizzes (mc/tf/sa, leveled) + final test
        │
[7] Source Verification (INDEPENDENT) ──► SOURCE_PAPER + every citation resolves  ── GATE: zero unverified
        │
[8] Codegen / Build ──► content.js, glossary.js, source.js (exact engine contract) + de-topic strings
        │
[9] QA / Verifier (INDEPENDENT, code sandbox) ──► pass/fail report  ── GATE: Definition of Done (§10)
        │
[10] Deploy ──► env vars set ▸ vercel --prod ▸ verify EXACT printed Production URL ▸ smoke-test /api
```

Orchestrator owns: the Brief, routing, the three gates, bundle assembly, and deploy. Nothing
advances past a gate without it passing.

## 4. The Course Brief (intake contract)

The 8 UX steps compile to this. It is the only thing every downstream agent reads. Persist it as
`brief.json` in the deck folder.

```jsonc
{
  "meta": {
    "title": "",                       // step 2 — masterclass name
    "slug": "",                        // kebab-case; becomes the Vercel project + folder name
    "created": "ISO-8601",
    "engine_contract": "v-texas"       // which engine globals/contract this targets (§1 caveat)
  },

  "class_tier": {                      // quality tier — drives source floors, slide floors, and the
    "level": "professional"            // generation gate. One of: briefing | standard | professional | expert.
  },                                   // floors: briefing 4/1, standard 8/2, professional 12/3, expert 18/5
                                       // (usable-sources / primary-sources). professional is the default bar.

  "knowledge_base": {                  // step 3
    "uploads": [ {"path":"", "type":"pdf|docx|url|paste", "trust":"primary|secondary|unknown"} ],
    "research": {
      "owner": "creator",              // WHO sources the knowledge base — the primary control:
                                       //   creator  = the class maker supplies sources (default)
                                       //   assisted = AI helps research; creator still approves
                                       //   ai       = Bernard runs web-search discovery + verification
      "mode": "grounded",              // descriptive sub-setting: none | grounded | collaborative
                                       // (grounded = corpus-only; collaborative = AI may web-research)
      "seed_prompts": [],              // what to research / questions to answer
      "allow_web": true,
      "recency_floor": "2024-01-01"    // reject stale facts older than this unless historical
    },
    "credibility": {
      "min_tier": "secondary",         // gate: claims below this tier get flagged, not shipped
      "require_two_sources_for": ["statistics","forward-looking claims","contested points"]
    }
  },

  "objectives": {                      // step 4 — AI-assisted, human-approved
    "terminal": [],                    // "by the end, a learner can …"  (3–7)
    "enabling": [],                    // sub-skills feeding the terminals
    "out_of_scope": []                 // explicit exclusions (prevents scope creep)
  },

  "mastery": {                         // step 5 — extent & granularity
    "target_level": 3,                 // 1 Novice · 2 Conversational · 3 Proficient · 4 Mastery · 5 SME/Teach
    "granularity": "survey | working | deep",
    "deep_dive_density": "low|med|high",   // how many slides get a `paper`
    "field_disagreement": true         // include "Where the field disagrees" named-opposition blocks
  },

  "audience": {                        // step 6 — design to BOTH average and floor
    "average": { "age_band":"", "education":"", "background":"", "technical":"non|mixed|technical",
                 "role":"IC|supervisor|manager|exec|labor|student|mixed" },
    "floor":   { "age_band":"", "education":"", "background":"", "technical":"non|mixed|technical",
                 "role":"" },          // lowest-common-denominator: reading level + assumed-knowledge floor
    "gender_mix": "",                  // informs examples/tone only; never gates content
    "tone": "plain | warm | executive | academic | workshop",  // must match brief-validator ENUMS
    "accessibility": { "reading_grade_cap": 9 }   // hard ceiling derived from `floor`
  },

  "length": {                          // step 7
    "minutes": 60,
    "slide_budget": 90,                // derived; Orchestrator computes from minutes × cadence
    "interaction_budget": { "polls": 2, "word_clouds": 4, "quizzes": 1, "final_test": true }
  },

  "language": {                        // step 8
    "primary": "en",                   // BCP-47; all generated prose + UI strings localize to this
    "localize_ui_strings": true,       // tutor greeting, select-text prompts, chat desc, etc.
    "glossary_in_primary": true
  }
}
```

**Derivations the Orchestrator computes from the Brief (so agents don't have to):**
- `slide_budget ≈ minutes × 1.5` (tune to cadence), minus interaction slides.
- `reading_grade_cap` from `audience.floor.education` → hard cap passed to every authoring agent.
- Quiz `level` gating ceiling = `mastery.target_level`.
- Section count + per-section slide allocation (Curriculum agent refines).

## 5. Agent roster

Each entry is a drop-in spec: **Role · Inputs · Outputs · Hard rules · Done when.** Copy-paste
prompt bodies in §6.

| # | Agent | Independent? | Primary tools |
|---|---|---|---|
| 0 | **Orchestrator / Producer** | — | filesystem, shell (deploy), the three gates |
| 1 | **Intake / Brief** | no | the wizard UI (or a prompt) → `brief.json` |
| 2 | **Knowledge Base / Research** | no | file parsing, web search/fetch, retrieval |
| 3 | **Curriculum / Learning Design** | no | corpus, Brief |
| 4 | **Content Author** (×N, section-parallel) | no | corpus, lesson plan |
| 5 | **Glossary** | no | finished slide/paper text |
| 6 | **Assessment** | no | corpus, lesson plan, level plan |
| 7 | **Source Verification** | **YES** | web fetch, the corpus provenance |
| 8 | **Codegen / Build** | no | deterministic emitter (the generator) |
| 9 | **QA / Verifier** | **YES** | Node sandbox (parse + count) |
| 10 | **Deploy** | no | shell (`vercel --prod`), endpoint smoke test |

**0 · Orchestrator** — Owns the Brief and the bundle. Routes each stage, blocks on gates, assembles
a **flat zip** (files at root, `api/` as a subfolder), runs Deploy. Never authors content itself.
*Done when:* live URL passes smoke tests.

**1 · Intake/Brief** — Turn the 8 steps into a *valid* `brief.json`. AI-assists objectives (step 4)
and proposes slide/interaction budgets. *Hard rules:* validate against the schema; flag missing
`floor` audience data (don't guess it). *Done when:* schema-valid Brief, human-approved.

**2 · Knowledge Base / Research** — Build a **Cited Corpus**: every usable claim tagged with source
+ tier + date. Three modes: ingest uploads only; *grounded* research (answer seed prompts strictly
from the corpus, NotebookLM-style); *collaborative* (may web-research and propose new sources).
*Hard rules:* every claim carries provenance; apply the credibility gate (`min_tier`,
two-sources-for); never invent a source; mark gaps as gaps. *Done when:* corpus covers all
`objectives.enabling`, credibility gate passed. **GATE.**

**3 · Curriculum / Learning Design** — From corpus + objectives + mastery + audience + length →
a **Lesson Plan**: ordered sections, slide budget per section, the *interaction map* (which slide
ids carry `poll`/`words`/`quiz`/`paper`, and which carry tiles / zoomable diagrams / heatmaps), and
the *level plan* (which questions exist at levels
1–5 up to `mastery.target_level`). *Hard rules:* respect `interaction_budget` and `slide_budget`;
every terminal objective maps to ≥1 assessment item; place a real **Works-Cited** slide near the
end. *Done when:* plan accounts for the full slide budget and all objectives.

**4 · Content Author** (section-parallel) — Write each slide's `deck` innerHTML and, where the plan
says: a single-object `paper` deep dive; **tile cards** (`data-title/data-more/data-dd/data-src`);
**zoomable diagrams** (the diagram markup `bindDiagrams` lights up); **heatmap tables**
(cells classed `tone-good/tone-mid/tone-bad/tone-hi/tone-lo`); and "Where the field disagrees"
named-opposition blocks. *Hard rules:* stay under `reading_grade_cap`; tune examples to `audience`
+ `tone`; write in `language.primary`; ground every factual sentence in the corpus and attach
citations (`<sup class="cite" data-src="sN">`); `paper` is a **single object** `{secnum,h,body}` —
never an array; deep-dive buttons `<button class="deepbtn" data-deep="slide-id">`. *Done when:* every
planned slide exists with valid innerHTML, every tile/diagram/heatmap the plan calls for is present,
and all citations resolve.

**5 · Glossary** — Extract terms *actually used* in finished slides/papers → plain-language defs at
the audience floor's reading level. *Hard rules:* define only terms present in the deck; longest
terms first (engine matches greedily). *Done when:* `GLOSSARY` covers jargon a floor-level learner
would stumble on.

**6 · Assessment** — Author quizzes + final test from the level plan. Types: `mc` (`options`,
`answer` idx, `why`), `tf` (`answer` bool, `why`), `sa` (`rubric`, `sample`, `accept[]`). *Hard
rules:* schema keys exactly `options/answer/why/type` (not `opts/a`); each question gets a `level`
1–5; `sa` items get `accept[]` keyword fallback for offline grading; embed as
`data-quiz='[…JSON…]'` (pop quizzes `data-pop="1"`, final test omits it). *Done when:* every
`data-quiz` JSON parses; every terminal objective is assessed.

**7 · Source Verification** *(INDEPENDENT)* — Independently re-check every citation against the real
source; build `SOURCE_PAPER = {title, cite, sections:[{id,num,title,body}]}`. *Hard rules:* a citation that doesn't resolve to a
real, locatable source is **removed and its claim softened or cut** — never papered over; the in-deck
"Student Reader" is a study aid and must be labeled as not original scholarship; the Works-Cited
slide lists only verified sources. *Done when:* zero unresolved `data-src` references; zero
fabricated sources. **GATE.**

**8 · Codegen / Build** — Deterministically emit `content.js` (`CLASS_TITLE`, `SLIDES`, `POLLS`,
`WORDS`), `glossary.js`, `source.js` to the exact engine contract; de-topic the shell + backend
strings (§6 checklist). *Hard rules:* emit to the globals the shipped engine actually reads (§1
caveat); preserve `index.html` UTF-8 BOM + CRLF if editing it (byte-level writes, not PowerShell
`Set-Content -Encoding UTF8`); keep `openPaper` global; load order
`content → glossary → source → engine → navscrubber`. *Done when:* files emitted and parse.

**9 · QA / Verifier** *(INDEPENDENT, code sandbox)* — Run the harness in §7. *Done when:* every
check in the Definition of Done (§10) passes. **GATE.**

**10 · Deploy** — Set the 5–6 env vars on the deck's *own* Vercel project; `vercel --prod`; trust
**only the Production URL it prints** (not "Ready in 9s"); smoke-test `/api/chat`, `/api/grade`,
`/api/tts`, `/api/poll`, `/api/words`. *Hard rules:* one Vercel project per deck; a bare prompt with
no printed URL = failed deploy; `429 insufficient_quota` = no billing on that key's org. *Done when:*
all endpoints answer 200 on the printed URL.

newer engine. This is why Phase 0 (freeze the contract) comes first.

## 6. Reusable prompts (the scaffold)

### 6a. Master orchestrator prompt (system-level)

```
You are the Producer of a Masterclass Factory. You turn a Course Brief (brief.json) into a
deployable interactive masterclass that plugs into a FIXED engine. You never edit the engine,
shell, or backends except a short list of topic strings. You only generate the CONTENT layer:
content.js (SLIDES/POLLS/WORDS), glossary.js (GLOSSARY), source.js (SOURCE_PAPER).

Run this pipeline in order, blocking on gates:
1 Brief → 2 Research(GATE: credibility) → 3 Curriculum → 4 Author ‖ 5 Glossary ‖ 6 Assessment →
7 Source Verification(GATE: zero unverified) → 8 Codegen → 9 QA(GATE: Definition of Done) → 10 Deploy.

Rules that never bend:
- Content is data, not code. Emit to the exact globals the shipped engine reads; verify them first.
- `paper` is a SINGLE object {secnum,h,body}, never an array.
- POLLS/WORDS are deck-defined; emit window.POLLS/window.WORDS keyed to the ids slides reference.
- Quiz keys are options/answer/why/type. Short-answer items carry rubric+sample+accept[] and a level.
- Every factual sentence is grounded in the corpus and cited; unverifiable claims are cut, not faked.
- Stay under the audience floor's reading-grade cap; write in the Brief's primary language.
- De-topic every hardcoded string (checklist 6e). Ship a FLAT zip. Trust only the printed Production URL.

Before any stage, restate the inputs you believe you have and what you will produce. After each
stage, report what passed/failed against that stage's Done-criteria. Do not advance past a gate
that did not pass.
```

PRODUCE: <the exact artifact + format>

### 6c. Source Verification prompt (the strict one)

```
ROLE: Independent Source Verifier. You did NOT write this content; assume it is wrong until proven.
READ: every <sup class="cite" data-src="sN"> in the deck and the corpus provenance for each.
TASK: for each citation, independently confirm the source exists and actually supports the claim.
OUTPUT: source.js (window.SOURCE_PAPER = {title, cite, sections:[{id,num,title,body}]}) containing ONLY verified sources, plus a
report listing: (a) verified, (b) claim-not-supported (soften/cut the claim), (c) source-not-found
(remove the citation and the claim it props up).
HARD RULES: never fabricate a source, a title, a date, or a URL. If you cannot locate it, it does
not exist for our purposes. The Student Reader is a study aid, not scholarship — say so in the deck.
DONE WHEN: zero data-src references are unresolved and zero sources are unverified.
```

[ ] content.js  — window.CLASS_TITLE

ladder (Novice→SME) is topic-agnostic — reuse it verbatim.

## 7. QA / Verifier harness (runnable)

The QA gate is **code, not prose.** Run this (or its equivalent) in a sandbox after codegen. It
parses the generated files with no browser and asserts the Definition of Done.

```js
// qa.js — run: node qa.js   (exits non-zero on any failure)
global.window = {};
require("./content.js"); require("./glossary.js"); require("./source.js");
const S = window.SLIDES || [], POLLS = window.POLLS || {}, WORDS = window.WORDS || {};
const PAPER = window.SOURCE_PAPER || null;
const SECTIONS = PAPER && Array.isArray(PAPER.sections) ? PAPER.sections : [];
const G = window.GLOSSARY || {};
const fail = [], warn = [];
const A = (cond, msg) => { if (!cond) fail.push(msg); };

A(S.length > 0, "no slides");
A(window.CLASS_TITLE, "missing CLASS_TITLE");

const ids = new Set();
S.forEach((s, i) => {
  A(s.id, `slide ${i} missing id`);
  A(!ids.has(s.id), `duplicate id: ${s.id}`); ids.add(s.id);
  A(typeof s.deck === "string" && s.deck.includes("<"), `slide ${s.id} deck not innerHTML`);
  if (s.paper) A(!Array.isArray(s.paper) && s.paper.h && s.paper.body,
                 `slide ${s.id} paper must be a single object {secnum,h,body}`);
  if (s.poll)  A(POLLS[s.poll] && Array.isArray(POLLS[s.poll].opts),
                 `poll id "${s.poll}" (slide ${s.id}) not defined in POLLS or missing opts`);
  if (s.words) A(WORDS[s.words] && WORDS[s.words].q,
                 `words id "${s.words}" (slide ${s.id}) not defined in WORDS`);
});

// every data-quiz JSON parses; keys + types are correct
let quizzes = 0, deepdives = 0;
S.forEach(s => {
  if (s.paper) deepdives++;
  const m = (s.deck.match(/data-quiz='([^']+)'/g) || []);
  m.forEach(raw => {
    quizzes++;
    let json = raw.slice(11, -1).replace(/&quot;/g, '"').replace(/&#39;/g, "'").replace(/&amp;/g, "&");
    let qs; try { qs = JSON.parse(json); } catch (e) { return fail.push(`bad quiz JSON in ${s.id}: ${e.message}`); }
    qs.forEach((q, j) => {
      A(["mc","tf","sa"].includes(q.type), `${s.id} q${j} bad type "${q.type}"`);
      if (q.type === "mc") A(Array.isArray(q.options) && Number.isInteger(q.answer), `${s.id} q${j} mc needs options[]+answer idx`);
      if (q.type === "tf") A(typeof q.answer === "boolean", `${s.id} q${j} tf needs boolean answer`);
      if (q.type === "sa") A(q.rubric && Array.isArray(q.accept), `${s.id} q${j} sa needs rubric+accept[]`);
      if (q.level) A(q.level >= 1 && q.level <= 5, `${s.id} q${j} level out of 1–5`);
    });
  });
});

// every citation in prose resolves to a real section id in SOURCE_PAPER.sections[]
const srcIds = new Set(SECTIONS.map(x => String(x.id)));
S.forEach(s => {
  (s.deck + (s.paper ? s.paper.body : "")).replace(/data-src="(s?\w+)"/g, (_, id) => {
    A(srcIds.has(id) || srcIds.has(id.replace(/^s/, "")), `unresolved citation ${id} in ${s.id}`); return _;
  });
});

A(SECTIONS.length > 0, "no SOURCE_PAPER.sections (Works Cited empty)");
A(!PAPER || (PAPER.title && PAPER.cite), "SOURCE_PAPER missing title/cite");
// GLOSSARY terms map to {d,r} objects (not strings)
A(Object.keys(G).length > 0, "glossary empty");
Object.keys(G).forEach(k => A(G[k] && typeof G[k] === "object" && G[k].d && G[k].r,
  `glossary "${k}" must be {d,r} (definition + why-it-matters)`));

console.log(`slides=${S.length} deepdives=${deepdives} quizzes=${quizzes} sections=${SECTIONS.length} glossary=${Object.keys(G).length}`);
if (warn.length) console.warn("WARN:\n - " + warn.join("\n - "));
if (fail.length) { console.error("FAIL:\n - " + fail.join("\n - ")); process.exit(1); }
console.log("QA PASS");
```

> Adapt the HTML-entity unescaping to however your codegen escapes `data-quiz`. Add checks as you
> discover new failure modes — this harness *is* your Definition of Done.

**Behaviors that bit earlier builds — now confirmed in code:**
- **TTS has NO server-side `tts-1` fallback.** Despite comments to the contrary (in the file *and*
  the companion engine doc), the function calls `gpt-4o-mini-tts` only and returns **502** on any
  OpenAI error. The *single* fallback path is **missing key → 503 → the client uses the browser
  voice**. So "robotic voice" = the key is missing or unfunded (503), **not** a model downgrade. A
  `429 insufficient_quota` returns 502 and the client should treat it as TTS-down.
- **Word input is sanitized server-side:** lowercased, stripped to `[a-z0-9- ]`, capped at 24 chars.
  Author word-cloud prompts expecting short single-token answers.
- **`grade.js` BAR{} ladder (levels 1–5, Novice→SME/Teach) is topic-agnostic** — copy it verbatim
  into every deck; only the topic + audience-level + sensitivity lines in its `sys` change.
- **Poll/words clients fall back to `localStorage`** when the store is unreachable (503), so the
  modals still work offline — but tallies won't be shared.

## 9. Build plan (phased)

**Phase 0 — Freeze the contract (½ day).** Pick the engine version; run the `grep` from §1 and
record the exact globals (the engines read `SOURCE_PAPER`, NOT `SOURCES`); standardize on deck-defined
`POLLS`/`WORDS` and `SOURCE_PAPER`. Write the `brief.json` schema (§4) into the repo. *Output:* a frozen
contract + an empty Brief template.

**Phase 1 — Manual-loop MVP (1–2 days).** No platform. You hand-fill a Brief, then run agents 2–9 as
*sequential prompts* in ChatGPT/Codex, assemble by hand, run `qa.js`, `vercel --prod`. Prove **one new
deck end-to-end** against the engine. This validates the prompts and the QA harness before any
automation. *Output:* a second live deck built only from this doc.

**Phase 2 — Orchestrate (3–5 days).** Turn the roster into Claude Code subagents/skills (or an API
script that chains the stages). Automate the three gates: credibility, source-verification,
`qa.js`. Add section-parallel authoring. *Output:* `producer` command that takes `brief.json` → live
URL with gates enforced.

**Phase 3 — Platform UI (1–2 weeks).** Build the 8-step wizard front-end that writes `brief.json`;
a job runner that invokes the orchestrator; per-deck Vercel project provisioning; a settings panel
(model choices, voice, credibility tier). *Output:* "Create a masterclass" button → deck.

**Phase 4 — RAG + KB hardening (ongoing).** Persistent knowledge base ingestion (PDF/DOCX/URL/paste/
web research), RAG-aware authoring (authors retrieve from the corpus instead of being handed it),
hybrid credibility gate, lesson-plan editor. *Output:* the full Factory.

> Don't skip Phase 1. Building the platform before one deck has been produced by hand from these
> prompts is the fast path to discovering your contract was wrong at the worst time.

## 10. Definition of Done (gate checklist)

**Content & contract**
- [ ] Every slide has a unique `id`; joins/patches are by `id`, never title or position.
- [ ] `paper` is a single object `{secnum,h,body}` everywhere (no one-element arrays).
- [ ] `window.POLLS`/`window.WORDS` defined and keyed to every `poll`/`words` id slides reference.
- [ ] Quiz JSON parses; keys are `options/answer/why/type`; `sa` items have `rubric`+`accept[]`;
      every question has a `level` ≤ `mastery.target_level`.
- [ ] Every `data-src` citation resolves to a verified source section in `SOURCE_PAPER.sections`.
- [ ] Glossary scans deep dives too (`#paperInner`, selector `p, .mini`) — verify in the engine.
- [ ] Reading grade ≤ audience floor cap; everything in `language.primary`.

**Sources (the hard gate)**
- [ ] Real Works-Cited slide near the end with **verified** sources only — zero fabricated/unverified.
- [ ] In-deck "Student Reader" is labeled a study aid, not original scholarship.

**De-topic**
- [ ] All strings in the §6e checklist replaced; no leftover prior-topic references.

**Build & bundle**
- [ ] Emitted to the globals THIS engine reads; `openPaper` stays global.
- [ ] `index.html` UTF-8 BOM + CRLF preserved (byte-level write; never PS `Set-Content -Encoding UTF8`).
- [ ] Load order `content → glossary → source → engine → navscrubber`.
- [ ] Ship a **flat zip** (files at root, `api/` subfolder) — no zip-in-a-folder nesting.
- [ ] No stray duplicate files (e.g. a second `chat.js` at root); one canonical `content.js`.

**Deploy**
- [ ] 4–5 env vars set on the deck's own Vercel project (`OPENAI_API_KEY`, optional
      `OPENAI_MODEL`, `KV_REST_API_URL`, `KV_REST_API_TOKEN`, `POLL_ADMIN_KEY`).
- [ ] Deployed via `vercel --prod`; success = a printed **Production URL** (not "Ready in 9s").
- [ ] Smoke-test `/api/chat`, `/api/grade`, `/api/tts`, `/api/poll`, `/api/words`, `/api/quality` on the printed URL.
- [ ] `/api/chat` model is a current OpenAI model string; default is `gpt-5.5`.
- [ ] Listen mode: `/api/tts` returns mp3. NO server-side tts-1 fallback exists — robotic voice
      means the key is missing/unfunded (503 → browser voice) or OpenAI errored (502, e.g. `429
      insufficient_quota`). Prime `speechSynthesis` on the user gesture; chunk browser voice into sentences.

## 11. Tool-specific notes

**Claude Code.** Make agents 2–10 **subagents** (or skills) and the Orchestrator the top-level
agent. Source Verification and QA get their *own* subagents so they run in independent contexts.
`qa.js` runs in the bash sandbox. The Orchestrator drives `vercel --prod` and reads back the printed
URL. The `brief.json` lives in the deck folder as the shared state.

**Codex / ChatGPT.** Run the roster as **ordered prompt stages**, pasting each prior artifact into
the next. Keep Source Verification and QA as *separate sessions* so they don't inherit the authoring
context. Use the §7 harness verbatim — paste the generated files in and run it (or run locally).

**API orchestration (any).** One script: each stage is a call; gates are assertions that throw and
halt; authoring stages fan out by section and join. Persist `brief.json` + artifacts to disk between
calls (no shared memory across calls — pass everything each time).

**Across all four:** this document is the contract. When a model proposes editing the engine, shell,
or a backend beyond the §6e strings — stop it. The Factory only generates the content layer.
